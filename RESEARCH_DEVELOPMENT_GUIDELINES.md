# AI 调研项目开发规范（aimusic）

---

## 0. TL;DR（强制执行规则）

1. 一个独立业务调研需求，优先对应一个独立模块；同模块内的子能力调研可继续在模块内拆分
2. 立项阶段至少产出：PRD / TASKS / ARCHITECTURE；形成可复用能力后补齐接入文档 / 设计文档
3. 对外层只暴露：入口 + 配置 + 必要模型；`api/` 根目录优先只放入口和配置，必要模型统一放 `api/model/`
4. 调试能力必须可开关，不默认暴露；调试工程可临时默认开启，但必须在文档中明确说明
5. app 只负责宿主级入口与编排，不承载独立业务实现
6. 每个阶段必须可验收
7. 文档必须随阶段推进同步补齐
8. 模板用于提效，可按需裁剪；不适用的章节可以删除，阶段未到的文档可以后补

---

## 1. 文档目标

本规范用于统一 aimusic 项目中的调研开发方式。

目标：

- 统一调研模块的立项、设计、实现、验收和交付方式
- 避免需求分散、代码边界混乱、文档风格不一致
- 提升调研成果复用率，便于沉淀回正式项目
- 用尽量轻量的标准提升效率，而不是为了规范增加额外负担

一句话原则：

> 一个独立调研需求，优先对应一个独立功能模块，并输出一套可验收、可复用、可接入的完整成果

---

## 2. 适用范围

适用于：

- 页面交互调研
- 播放器能力调研
- 缓存能力调研
- 清理 / 扫描 / 导出 / 上传 / 分享等独立能力
- 新技术方案验证

不适用于：

- app 宿主导航调整
- 全局工程配置修改

---

## 3. 调研项目总原则

### 3.1 模块独立

独立业务调研需求，优先独立模块实现。

同一业务域内的小范围策略验证、能力补充或子能力调研，不强制新建 Gradle module，可以在现有模块内继续拆分：

- 独立 `feature-name/`
- 独立 `api/` 和 `internal/`
- 独立 `docs/`

目的：

- 边界清晰
- 易于 review
- 易于迁移

---

### 3.2 成果完整

每个模块最终应输出完整成果，而不是仅代码。

建议按阶段拆分：

- 立项和实现初期：`PRD / TASKS / ARCHITECTURE`
- 形成最小闭环后：可运行代码 + 阶段验收结果
- 形成可复用能力或稳定 API 后：接入文档 + 设计文档

最终完整成果包括：

- PRD
- TASKS
- ARCHITECTURE
- 接入文档
- 设计文档
- 可运行代码

---

### 3.3 边界清楚

必须区分：

- 对外层（API）
- 对内层（实现）

要求：

- 对外层简洁稳定
- 不暴露内部调度与策略
- 内部细节收敛到 internal
- 对接入方只提供“怎么用”所需的最小集合
- 对维护者保留“怎么实现”所需的完整细节

---

### 3.4 渐进实现

默认分阶段推进：

1. 搭结构
2. 最小闭环
3. 状态联动
4. 调试能力
5. API 收口
6. 文档补齐

---

## 4. 推荐研发流程

1. 明确调研目标
2. 判断是新建独立模块，还是在现有模块内新增独立子能力
3. 输出 PRD.md
4. 输出 ARCHITECTURE.md
5. 输出 TASKS.md
6. 实现最小闭环
7. 分阶段验收
8. 能力形成稳定入口后，输出接入文档
9. 内部方案稳定后，输出设计文档
10. 决定是否沉淀到正式项目

---

## 5. 标准输出物

### 5.1 PRD.md

定义：

- 背景
- 调研目标
- 功能结构
- 关键交互
- 数据范围
- 宿主接入方式
- 验收标准
- 当前限制

---

### 5.2 TASKS.md

定义：

- 当前状态
- 分阶段计划
- 每阶段目标
- 验收标准

---

### 5.3 ARCHITECTURE.md

定义：

- 模块定位
- 目录结构
- 分层关系
- 宿主边界
- 演进方向

---

### 5.4 接入文档（API_GUIDE / QUICK_START）

面向接入方：

- 入口类
- 配置类
- 必要模型
- 使用步骤
- 初始化方式
- 生命周期或调用时机
- 默认行为
- 注意事项

禁止：

- 依赖内部实现理解
- 要求接入方先读内部代码才能使用

---

### 5.5 设计文档（DESIGN.md）

面向维护者：

- 分层结构
- 流程图
- 调度逻辑
- 策略规则
- 边界说明
- 关键类职责
- 风险点与降级策略

---

### 5.6 文档职责划分

| 文档 | 作用 |
|------|------|
| PRD | 做什么 |
| TASKS | 怎么分阶段做 |
| ARCHITECTURE | 怎么分层 |
| 接入文档 | 怎么用 |
| 设计文档 | 怎么实现 |

快速判断规则：

- 别人怎么接入我，写进接入文档
- 我内部怎么实现，写进设计文档
- 这轮需求要做什么，写进 PRD 和 TASKS
- 当前做到哪一阶段、是否可验收，写进 TASKS 或阶段验收记录

---

### 5.7 调研完成后的标准成果包

每个调研需求在阶段完成或调研结束后，建议至少沉淀出一份标准成果包，方便后续快速回收和接入正式项目。

成果包建议包含：

- 需求结论：这次调研验证了什么，结论是什么
- 代码入口：模块入口类、配置类、关键模型
- 接入方式：宿主如何进入，业务如何调用
- 验收结果：哪些能力已完成，哪些仍是限制项
- 风险说明：当前不建议直接落正式项目的部分
- 后续建议：下一阶段该继续补什么

如果需要进一步沉淀到正式项目，接收方只需要先看：

1. PRD
2. 接入文档
3. TASKS 中的当前状态 / 已完成阶段
4. 设计文档中的风险与边界

---

## 6. 标准目录约定

每个调研模块推荐遵循以下结构：

```text
module-name/
  src/main/java/...
  src/main/res/...
  docs/
    PRD.md
    TASKS.md
    ARCHITECTURE.md
```

如存在子能力拆分：

```text
module-name/
  src/main/java/.../feature-name/
    api/
    internal/
  docs/
    PRD.md
    TASKS.md
    ARCHITECTURE.md
    FEATURE_DESIGN.md
    FEATURE_INTEGRATION.md
```

如果不想每次从空白文档开始写，可以直接复用项目中的模板：

- `templates/module-docs/PRD_TEMPLATE.md`
- `templates/module-docs/TASKS_TEMPLATE.md`
- `templates/module-docs/ARCHITECTURE_TEMPLATE.md`
- `templates/module-docs/INTEGRATION_TEMPLATE.md`
- `templates/module-docs/DESIGN_TEMPLATE.md`

重要说明：

- 模板是为了提效，不是要求每个模块都机械补齐所有章节
- 不适用的章节可以直接删除
- 当前阶段暂时不需要的文档，可以在阶段推进后再补
- 小型调研优先保留最小必需信息，不要为了“完整”增加维护成本

---

## 7. 模块设计规范

### 7.1 模块定位

说明：

- 调研目标
- 验证能力
- 与宿主边界

---

### 7.2 对外接口（API）

包括：

- Entry
- Config
- Model（Request / Result / Snapshot）

命名建议：

- Feature
- ModuleEntry
- Config
- Snapshot

要求：

- 简洁
- 稳定
- 不暴露 internal
- 能让接入方只通过入口类和配置类完成接入
- 默认值清楚，必要配置收敛
- `api/` 根目录优先只放入口类、配置类和少量确实需要直接调用的门面类
- `api/model/` 统一放对外输入、输出、快照等契约模型
- 如果模型只是内部流转对象，不要放进 `api/model/`，应继续收口在 `internal`

推荐口径：

- 一个入口类：负责初始化、生命周期衔接、对外调用
- 一个配置类：负责可调参数、debug 开关、策略开关
- 少量必要模型：负责输入输出，不扩散内部状态对象

目录建议：

```text
api/
  XxxFeature.kt
  XxxConfig.kt
  model/
    XxxItem.kt
    XxxResult.kt
    XxxSnapshot.kt
internal/
  ...
```

判断规则：

- 接入方需要主动 new、配置或调用的类，优先放 `api/` 根目录
- 接入方只是作为参数、返回值或调试快照消费的模型，优先放 `api/model/`
- 只有模块内部实现需要感知的状态、策略、任务对象，放 `internal/`

---

### 7.3 对内实现

包括：

- manager
- scheduler
- strategy
- data source
- builder

要求：

- 放入 internal
- 不对外暴露
- 对外不要直接依赖具体 manager / scheduler / strategy

典型职责示例：

- `Manager`：承接能力协调
- `Scheduler`：承接任务调度
- `Strategy`：承接规则与策略选择
- `Builder`：承接快照、数据组装、配置转换

---

### 7.4 API 稳定性分级

- Stable：可用于正式项目
- Experimental：仅调研使用

建议通过注释或包结构标识

---

### 7.5 验收方式

必须明确：

- 是否跑通
- 状态是否联动
- 是否可接入
- 当前限制

建议额外明确：

- 宿主如何进入
- 外部最小接入代码是否已具备
- 调试能力是否已隔离
- 是否已具备迁移到正式项目的基础条件

---

## 8. app 宿主职责

app 只负责：

- 模块统一入口
- 宿主级导航
- 权限申请与权限编排
- Launcher 配置
- App Shortcuts
- 宿主级 deeplink 或统一跳转承接
- 调试入口聚合

app 不负责：

- 业务逻辑
- 页面实现
- 调研代码

一句话：

app 负责“怎么进、怎么统一承接”，模块负责“具体做什么”

---

## 9. 实现阶段要求

### 9.1 最小闭环优先

顺序：

1. 跑通
2. 联动
3. 边界
4. 调试
5. API
6. 文档

---

### 9.2 每阶段可验收

每阶段必须：

- 有目标
- 有边界
- 可验证

推荐每阶段都补一段简短验收记录，至少说明：

- 本阶段完成了什么
- 本阶段未做什么
- 现在能怎么验收
- 下一阶段要补什么

---

### 9.3 禁止过度抽象

允许：

- 先实现再收敛

但必须：

- 不混多个需求
- 不暴露调试逻辑

---

## 10. 调试能力规范

### 10.1 正式能力

- 可用于正式项目
- 稳定 API

---

### 10.2 调试能力

包括：

- 调试 UI
- 日志
- 快照
- 面板

---

### 强制规则（重要）

- 调试能力不得进入正式页面路径
- 必须通过 debug flag 显式开启
- 必须在文档中标明

调试工程例外说明：

- 如果当前工程本身就是调研 / 调试工程，允许为了观察效果临时默认开启调试能力
- 但必须在模块 PRD / TASKS / 接入文档中明确写清楚“当前工程默认开启，仅供调研使用”
- 一旦准备迁移到正式项目，默认值应回到关闭态

---

## 11. 接入文档规范

必须回答：

- 入口类是什么
- 配置类是什么
- 必要模型是什么
- 如何接入
- 默认配置
- 生命周期 / 调用时机
- 最小示例代码
- 注意事项

禁止：

- 依赖阅读内部代码
- 把内部流程图当成接入步骤

目标是让接入方读完接入文档后，能直接回答：

- 我应该依赖哪个入口类
- 我有哪些配置要传
- 默认情况下它会做什么
- 我需要在什么时机调用
- 出问题先看哪里

---

## 12. 设计文档规范

必须包含：

- 分层结构
- 流程图
- 调度逻辑
- 策略说明
- 风险与边界

建议补充：

- 关键类职责表
- 核心状态流转
- 失败场景和降级方案
- 为什么这样设计，而不是其他方案

---

## 13. 验收规范

必须检查：

- 功能可运行
- 状态正确
- 宿主可进入
- 文档齐全
- 构建通过
- 调试能力隔离

如果目标是“后续快速接入正式项目”，还应补充检查：

- 入口类和配置类是否已经收口
- 对外是否仍依赖内部实现类
- 默认配置是否可直接运行
- 接入文档是否足够让其他人独立完成接入

---

## 14. 最终交付标准

每个模块必须包含：

- 可运行代码
- 宿主入口
- PRD
- TASKS
- ARCHITECTURE
- 接入文档（已形成可复用入口时必须补齐）
- 设计文档（已形成明确内部方案时必须补齐）
- 构建验证结果
- 当前阶段结论与限制说明

如果目标是快速接入正式项目，最终还应满足：

- 对外入口已稳定
- 配置项已收口
- 默认接入链路可跑通
- 文档能支持他人独立接入

---

## 15. 模块命名规范

建议：

- aimusic-feature-xxx（可沉淀能力）
- aimusic-research-xxx（调研验证）

---

## 16. 模块演进阶段

- V0：调研验证
- V1：能力收敛
- V2：可接入正式项目

---

## 17. 迁移到正式项目标准

满足以下条件：

- API 已稳定
- 无调试依赖
- 核心链路稳定
- 文档完整
- 默认配置可直接工作
- 风险和限制已明确

---

## 18. 模块交付 Checklist（强制）

[ ] 独立业务调研需求已明确承载位置（独立模块或模块内独立子能力）  
[ ] PRD.md  
[ ] TASKS.md  
[ ] ARCHITECTURE.md  
[ ] 接入文档（形成可复用能力后补齐）  
[ ] 设计文档（内部方案稳定后补齐）  
[ ] 宿主入口  
[ ] 对外入口类、配置类、必要模型已收口  
[ ] 对外 / 对内分层清晰  
[ ] 无调试能力泄露  
[ ] 每阶段可验收  
[ ] 当前阶段结论与限制已记录  
[ ] 能支持后续快速接入项目

---

## 19. 反例（禁止）

❌ 多个调研需求塞进一个模块  
❌ 在 app 中写业务逻辑  
❌ 对外暴露 internal 类  
❌ 调试 UI 默认对用户可见  
❌ 接入文档依赖内部实现
❌ 对外直接暴露 manager / scheduler / strategy  
❌ 调研做完后没有交付结论和限制说明

---

## 20. 快速接入导向要求

如果一个调研模块的目标是“后续快速接入正式项目”，则在设计和交付时必须额外满足以下要求：

- 对外只有少量稳定入口，不要求接入方理解内部分层
- 配置项有默认值，开箱即可运行
- 文档优先讲“如何接入”，再讲“内部怎么做”
- 当前限制、风险和未完成项必须明确
- 调试能力与正式能力必须严格分开

推荐输出顺序：

1. 给出入口类
2. 给出配置类
3. 给出最小示例代码
4. 给出默认行为说明
5. 给出限制与风险说明

---

## 21. 推荐交付模板

调研完成后，建议至少用下面的结构做一次收束：

1. 调研目标与结论
2. 当前可运行能力
3. 对外入口、配置、模型
4. 宿主接入方式
5. 当前限制与风险
6. 下一阶段建议

---

## 22. 使用建议

新模块开发顺序：

1. 阅读本规范
2. 判断模块承载方式
3. 写 PRD
4. 写 ARCHITECTURE
5. 写 TASKS
6. 实现最小闭环
7. 做阶段验收
8. 收口入口类与配置类
9. 补接入文档
10. 补设计文档
11. 输出阶段结论

---

## 23. 规范目标总结

本规范最终希望每一个调研模块都具备以下特征：

- 可验收：每个阶段都有目标、边界和验证方式
- 可复用：调研结果不是一次性代码，而是可沉淀能力
- 可接入：外部通过入口类和配置类即可快速接入
- 可迁移：后续能较低成本回收到正式项目
- 可维护：内部实现、风险和边界都有文档可追溯
